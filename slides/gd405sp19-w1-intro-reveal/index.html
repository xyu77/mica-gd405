Turtle exercise

Think of a turtle at the center of paper.

You will create instructions for it to follow and move.

When the turtle moves, it leaves a trace.

I will give you a shape to draw.

Discuss as a group to decide what instructions to use.

You can only use sounds for your instructions.

No spoken words.

Describe how, not what to draw.

"Draw a big rectangle." - how big? what kind of rectangle?

"Move 5 units up."

Meta-design — describe how to draw.

Half the group will give instructions.
The other half will execute them.

Let's do it.

  TO SQUARE
    FORWARD 100
    RIGHT 90
    FORWARD 100
    RIGHT 90
    FORWARD 100
    RIGHT 90
    FORWARD 100
    RIGHT 90
  END

from Mindstorm by Seymour Papert

This exercise is inspired by the book, Mindstorm and LOGO programming environment by Seymour Papert.

Here, we see two simple commands for the turtle to perform. And we can give this series of instruction a name; SQUARE. So, anytime we call SQUARE "function", it will perform the same instructions. 

  TO SQUARE
    REPEAT 4
    FORWARD 100
    RIGHT 90
  END

from Mindstorm by Seymour Papert

And because the same forward/right commands are repeated, we can tell it to repeat it 4 times. 

  TO SQUARE:SIZE
    REPEAT 4
    FORWARD :SIZE
    RIGHT 90
  END

from Mindstorm by Seymour Papert

Now, instead of telling the turtle to draw the same square every time, we can abstract the distnce it moves and make it into a variable, SIZE. Then, we can call the SQUARE function with different numbers to draw squares in different sizes.

    Programming/design Concepts

  • Parameterize
  • Function
  • Modularize
  • Iteration

With this simple exercise, we can learn many important design concepts that are already familiar to software engineers.

1. Instead of saying 'draw a BIG square,' we parameterize the SIZE to respond to different conditons.

2. The way we command the turtle, different actions it can take is a function. We also abstract the function SQUARE.

3. Each instruction is simple; move or turn. We can break down the complex instruction into smaller pieces.

4. We can develop the logic and test and repeat to improve it. That is an iteration.
